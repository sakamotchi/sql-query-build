# テスト手順書：SQLエディタ起動処理改善

## テスト方針

- 操作手順での確認を優先する
- テストコードは操作で確認できない単体ロジックのみ記載する

---

## 1. 基本動作確認（ローカル接続）

### 前提条件
- ローカルの PostgreSQL または MySQL が起動していること
- 複数テーブル（10件以上）が存在するDBに接続設定済みであること

### 手順

1. `npm run tauri:dev` でアプリを起動する
2. SQLエディタ画面を開く
3. ローカルDBの接続を選択する

### 確認項目

| # | 確認内容 | 期待結果 |
|---|---------|---------|
| 1 | 画面上部の進捗表示が出ること | "Loading columns: X/Y tables" が表示される |
| 2 | 進捗が完了すること | 進捗表示が消える（エラーにならない） |
| 3 | SQL補完でカラム名が表示されること | テーブル名の後に `.` を入力するとカラム候補が出る |
| 4 | サイドバーのテーブルツリーにカラムが表示されること | テーブルを展開するとカラム一覧が出る |

## 2. 高レイテンシ環境での動作確認（AWS RDS ポートフォワード）

### 前提条件
- AWS SSM または SSH ポートフォワードで RDS に接続できること
- 接続設定がアプリに登録済みであること

### 手順

1. ポートフォワードを開始する（例: `aws ssm start-session ...`）
2. アプリのSQLエディタでRDS接続を選択する
3. 進捗表示の速度を確認する

### 確認項目

| # | 確認内容 | 期待結果（改善後） |
|---|---------|----------------|
| 1 | 進捗の更新単位 | スキーマ単位でまとめて進捗が進む（以前はテーブル1件ずつ） |
| 2 | 全テーブル取得完了までの時間 | 改修前と比較して大幅に短縮されていること |
| 3 | エラーなく完了すること | タイムアウトや接続エラーが出ない |

### 補足：改修前後の比較方法

改修前後の時間を比較する場合、ブラウザの DevTools（Tauri では WebView の開発者ツール）でコンソールログを確認するか、
以下のログ出力を一時的にコードに追加して計測する：

```typescript
// startBackgroundFetch() の開始・終了時
const startTime = performance.now()
// ... 処理 ...
console.log(`[perf] Background fetch: ${performance.now() - startTime}ms`)
```

## 3. スキーマが複数ある場合の確認

### 手順

1. 複数スキーマ（例: `public`, `app`, `audit`）が存在するPostgreSQLに接続する
2. SQLエディタを起動して進捗表示を確認する

### 確認項目

| # | 確認内容 | 期待結果 |
|---|---------|---------|
| 1 | 全スキーマのテーブルが補完に出ること | 各スキーマのテーブル・カラムが補完される |
| 2 | システムスキーマ（`pg_catalog` など）が除外されること | システムスキーマのカラムは取得されない（進捗に含まれない） |

---

## 4. キャンセル・切断時の確認

### 手順

1. SQLエディタでDB接続を選択してバックグラウンド取得が始まったら、すぐに別の接続に切り替える

### 確認項目

| # | 確認内容 | 期待結果 |
|---|---------|---------|
| 1 | 前の接続のバックグラウンド処理が停止すること | コンソールに旧接続のログが出続けない |
| 2 | 新しい接続の取得が正常に開始されること | 新接続の進捗表示が始まる |

---

## 5. SQLiteでの動作確認

### 手順

1. SQLiteファイルの接続を選択してSQLエディタを開く

### 確認項目

| # | 確認内容 | 期待結果 |
|---|---------|---------|
| 1 | カラム補完が動作すること | テーブル名入力後にカラム候補が出る |
| 2 | エラーが出ないこと | コンソールにエラーログが出ない |
